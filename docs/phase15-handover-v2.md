# Backgammon Trainer — Phase 1.5 指示書 v2(アプリ統合編・最終版)

## ゴール

Phase 1で生成したデータセットを `index.html` に統合し、ver.2として完成させる。
ver.2のコンセプト: **「正解を当てるクイズ」から「損失の大きさを体感し、弱点を潰す練習システム」へ**。
ユーザーの最終目標はPR(1手あたり平均エクイティ損失)の改善であり、全タスクはこれに奉仕する。

## Task 0: オープニング整合性チェック(最重要・最初にやる)

背景: gnubg解析で、ロール64の最善手は `24/18 13/9` だったが、アプリの `OPENINGS_WHITE['64']` は `24/14`(差-0.01)。
この表はアプリ内で**2つの用途**に使われている:
1. オープニング暗記の正解表示
2. レスポンスクイズで「相手のオープニング後の盤面」を構築(`OPENINGS=OPENINGS_WHITE`、`applyGrayMove()`経由)

手順:
1. `scripts/generate_dataset.py` を確認し、**相手(グレー)の各オープニングにどのムーブを使ったか**を16キー全て特定
2. アプリの `OPENINGS_WHITE` と突き合わせる
3. 不一致キー(少なくとも64が疑わしい)は**gnubg最善手を正**として:
   - `OPENINGS_WHITE` を更新
   - 前提盤面が変わるキーは**該当オープニングの21問のみ再生成**
4. 再発防止として `response-quiz-v2.json` トップレベルに前提ムーブを埋め込む:
   ```json
   "openingMoves": { "65": "24/13", "64": "24/18 13/9", "...": "..." }
   ```

## Task 1: レスポンスクイズのJSON読み込み化

1. `PRESET_PROBLEMS`(ハードコード)を削除し、起動時に `./data/response-quiz-v2.json` を fetch
2. `moves` 配列(エクイティ順)→ 既存の `{choices:[{move}], answer:0}` 形式へ変換。**各選択肢のエクイティ差も保持**
3. `OPENINGS` テーブルとJSONの `openingMoves` を起動時に照合し、不一致は `console.warn`
4. fetch失敗時は「データ読み込み失敗。通信環境を確認して再読み込みしてください」を表示(`rUploadStatus` 流用可)
5. 既存の .txt アップロード機能は残す
6. 既存機能(選択肢シャッフル、フィルター、キーボード操作、反転モード、統計)はすべて維持

## Task 2: エクイティ表示 — 「損失の大きさ」を見せる

答え合わせ(`rConfirm()`)後の表示強化:

1. 各選択肢の右側にエクイティ差を表示(最善手は `BEST`、他は `-0.118` 形式)
2. 損失規模で色分け(既存CSS変数を使用):
   - `0.000`: `--correct` で `BEST`
   - `< 0.02`: ほぼ互角(dim系で数値のみ)
   - `0.02〜0.08`: エラー(`--perfect`=黄)
   - `>= 0.08`: ブランダー(`--wrong`=赤)
3. 不正解時はボードラベル部に「あなたの選択: -0.152」のような損失表示
4. **回答前にエクイティが見えてはいけない**(DOM上にも露出させない。data属性への格納も答え合わせ後に行う)

## Task 3: 弱点復習と損失統計(新規・ver.2の核)

レスポンスクイズを「解きっぱなし」から「弱点が回ってくる」構造にする:

1. **未正解フィルター**: 間違えた問題のIDを localStorage に保存し、フィルターバーに `MISSED n` ボタンを追加(オープニング暗記の既存実装 `oBuildFilter` の missed 方式を踏襲)。正解したらリストから外す
2. **平均損失の統計**: 従来の正答率に加えて、**回答の平均エクイティ損失**を記録・表示する
   - 保存: 既存 stats に累積損失と回答数を追加(localStorage キーは後方互換に注意。形式が変わる場合は新キーにする)
   - 表示: ヘッダーの統計表示を「正答率 + 平均損失」に(例: `82% / -0.031`)
   - これはユーザーのPRトレーニングの中核指標。正解でも僅差の次善手を選んだ場合の損失(=0ではない場合)も正しく計上する
3. 統計リセット手段がなければ簡易なものを追加してよい(例: 統計表示の長押し or ダブルタップで確認ダイアログ→リセット)

## Task 4: ピップカウント局面の実戦化

1. `./data/pip-positions.json` を fetch し、出題を実戦局面サンプリングに置き換え
2. 難易度フィルター(EASY/MEDIUM/HARD)を**フェーズフィルター**に変更: `RACE / HOLDING / PRIME / ALL`(bearinがデータにあれば追加)
3. 直近に出した局面の連続出題を避ける
4. fetch失敗時のフォールバックとして既存の `pGenPosition()` を残す
5. ピップ計算・判定ロジックは変更しない

## Task 5: Service Worker / PWA更新

1. `service-worker.js` の `ASSETS` に3つのJSONを追加
2. `CACHE_NAME` を `bg-trainer-v4` に更新
3. オンライン初回起動後はオフラインでも全機能が動作すること

- [ ] エクイティが回答前にDOM上にも露出していない
- [ ] MISSEDフィルター: 誤答→出現、正答→消える
- [ ] 統計に平均損失が表示され、リロード後も保持される
- [ ] フィルター、キーボード操作(A-D, Enter)、反転モード、.txtアップロードが動く
- [ ] オープニング暗記: 64の正解が更新されている
- [ ] ピップカウント: フェーズフィルターで実戦局面、判定・統計が動く
